# SOLID Principles & Design Patterns for C# / .NET

## SOLID Principles

### S - Single Responsibility Principle (SRP)
- A class should have only one reason to change
- Each class should have a single responsibility or job
- Separation of concerns
- Example: Separate data access, business logic, and presentation into different classes

### O - Open/Closed Principle (OCP)
- Software entities should be open for extension but closed for modification
- Add new functionality by adding new code, not changing existing code
- Achieved through inheritance, interfaces, and polymorphism
- Example: Use abstract classes or interfaces to define contracts, then extend with new implementations

### L - Liskov Substitution Principle (LSP)
- Derived classes must be substitutable for their base classes
- Objects of a superclass should be replaceable with objects of a subclass without affecting correctness
- Child classes should not break parent class behavior
- Example: If a method takes Animal, you should be able to pass Dog or Cat without issues

### I - Interface Segregation Principle (ISP)
- Clients should not be forced to depend on interfaces they do not use
- Split large interfaces into smaller, more specific ones
- Fat interfaces cause unnecessary dependencies
- Example: Instead of IWorker with Work() and Eat(), create IWorkable and IEatable

### D - Dependency Inversion Principle (DIP)
- High-level modules should not depend on low-level modules
- Both should depend on abstractions (interfaces)
- Abstractions should not depend on details; details should depend on abstractions
- Decouple components through dependency injection
- Example: Service layer depends on IRepository interface, not on SqlRepository directly

---

## Design Patterns

### Creational Patterns (Object Creation)

#### Singleton
- Ensures a class has only one instance and provides global access point
- Use for logging, configuration, connection pools, caching
- Thread-safe implementation with lazy initialization in C#

#### Factory Method
- Defines an interface for creating objects, but subclasses decide which class to instantiate
- Promotes loose coupling
- Example: Database factory returning SqlConnection or MySqlConnection

#### Abstract Factory
- Creates families of related objects without specifying concrete classes
- Use when system needs to be independent of product creation
- Example: UI factory creating Windows or Mac buttons, checkboxes, and dialogs

#### Builder
- Constructs complex objects step by step
- Separate construction from representation
- Example: Building a SQL query, HTTP request, or complex DTO

#### Prototype
- Creates new objects by cloning existing ones
- Avoids expensive initialization costs
- Example: Copying configuration objects, deep cloning complex entities

---

### Structural Patterns (Class/Object Composition)

#### Dependency Injection (DI) / Inversion of Control (IoC)
- Externalizes object creation and dependency management
- Common in ASP.NET Core (built-in DI container)
- Constructor injection, method injection, property injection

#### Repository
- Mediates between domain and data mapping layers
- Provides collection-like interface for data access
- Common with Entity Framework and ADO.NET

#### Unit of Work
- Maintains list of business objects affected by a transaction
- Coordinates writing out changes
- Often paired with Repository pattern

#### Adapter
- Converts interface of a class into another interface expected by clients
- Allows incompatible interfaces to work together
- Example: Adapting legacy data access code to new interface

#### Decorator
- Adds behavior to objects dynamically without affecting other objects
- Wraps an object with new functionality
- Example: Adding logging, caching, or validation to repository methods

#### Facade
- Provides simplified interface to complex subsystem
- Reduces dependencies and complexity
- Example: Service class that coordinates multiple lower-level classes

#### Proxy
- Provides placeholder for another object to control access
- Lazy loading, access control, logging, caching
- Example: Virtual proxy for expensive object initialization

#### Composite
- Composes objects into tree structures to represent part-whole hierarchies
- Clients treat individual objects and compositions uniformly
- Example: UI controls hierarchy, organizational structures

#### Bridge
- Decouples abstraction from implementation so both can vary independently
- Example: Remote control (abstraction) and device (implementation)

---

### Behavioral Patterns (Communication between Objects)

#### Strategy
- Defines family of algorithms, encapsulates each, and makes them interchangeable
- Allows algorithm to vary independently from clients
- Example: Payment processing (CreditCard, PayPal, Crypto), sorting strategies

#### Observer / Publish-Subscribe
- Defines one-to-many dependency where changes to one object notify dependents
- Events and delegates in C# are implementation of Observer
- Example: Event handlers, UI updates, message broadcasting

#### Command
- Encapsulates request as an object
- Supports undo/redo, queuing, logging
- Example: Menu actions, macro recording, task scheduling

#### Chain of Responsibility
- Passes request along chain of handlers until one handles it
- Decouples sender from receiver
- Example: Middleware pipeline in ASP.NET Core, logging levels

#### Template Method
- Defines skeleton of algorithm in method, deferring steps to subclasses
- Subclasses override specific steps without changing algorithm structure
- Example: Data processing base class with abstract Read/Process/Write methods

#### State
- Allows object to change behavior when internal state changes
- Appears as if object changed its class
- Example: Document with Draft, Moderation, Published states

#### Mediator
- Defines object that encapsulates how objects interact
- Reduces direct dependencies between objects
- Example: Chat room, control tower, form validation coordinator

#### Memento
- Captures and externalizes object's internal state without violating encapsulation
- Enables restore to previous state
- Example: Undo/redo, save game state, transaction rollback

#### Visitor
- Represents operation to be performed on elements of object structure
- Adds new operations without changing element classes
- Example: AST traversal, reporting across different document types

#### Interpreter
- Defines grammar representation and interpreter for language
- Example: SQL parsing, expression evaluation, configuration language

#### Iterator
- Provides way to access elements of aggregate object sequentially
- foreach loop and IEnumerable/IEnumerator in C#
- Example: Custom collection iteration

---

### Architectural Patterns

#### Model-View-Controller (MVC)
- Separates application into Model (data), View (UI), Controller (logic)
- ASP.NET Core MVC is built on this pattern

#### Model-View-ViewModel (MVVM)
- Similar to MVC but with ViewModel acting as mediator
- Common in WPF, Xamarin, MAUI, Blazor

#### Layered Architecture (N-Tier)
- Organizes code into Presentation, Business, Data Access layers
- 3-Tier Architecture: Presentation, Business Logic, Data Access

#### Microservices Architecture
- Decomposes application into small, independent services
- Each service has its own database and can be deployed independently

#### Event-Driven Architecture
- Components communicate through events
- Decoupled, asynchronous, scalable
- Example: Message queues (RabbitMQ, Azure Service Bus), Event Sourcing

#### CQRS (Command Query Responsibility Segregation)
- Separates read and write operations into different models
- Often used with Event Sourcing
- Improves scalability and performance

---

### Anti-Patterns to Avoid

- God Object (single class doing everything)
- Spaghetti Code (no structure, tangled dependencies)
- Golden Hammer (using familiar pattern for every problem)
- Boat Anchor (keeping unused code "just in case")
- Lava Flow (dead code nobody wants to remove)
- Copy-Paste Programming (code duplication)
- Magic Numbers (hard-coded values without explanation)
- Premature Optimization (optimizing before understanding bottlenecks)

---

### Patterns Specific to C# and .NET

- IDisposable Pattern (resource cleanup)
- Async/Await Pattern (asynchronous programming)
- Options Pattern (configuration binding in ASP.NET Core)
- Middleware Pattern (request processing pipeline)
- Hosted Service Pattern (background tasks)
- Result Pattern (explicit error handling without exceptions)
- Specification Pattern (encapsulated business rules as objects)
